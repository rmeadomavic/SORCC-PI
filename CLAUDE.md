# SORCC-PI вҖ” Claude Code Guidelines

## Project Context

SORCC-PI is the software toolkit for the **Special Operations Robotics Capabilities
Course (SORCC)** Module 4.3: Raspberry Pi Payload Integrator. It transforms a
Raspberry Pi 4 into a multi-capability RF survey payload that students mount on
robotics platforms вҖ” including 10" FPV quadcopters вҖ” for wireless reconnaissance
training exercises.

The primary training mission is an airborne WiFi hunt: students fly the payload
over an area to locate a known WiFi SSID, track its signal strength in real time,
and map the results in Google Earth after landing.

### Hardware Stack

| Component | Model | Purpose |
|-----------|-------|---------|
| SBC | Raspberry Pi 4 8GB | Main compute |
| OS | Kali Linux ARM64 | Base operating system |
| LTE | SixFab LE910Cx hat (Telit modem) | Cellular connectivity |
| Battery | PiSugar 5000mAh | Portable power |
| SDR | Nooelec SMART (RTL2832U) + antenna | RF reception (433MHz, ADS-B, etc.) |
| SIM | Carrier-configurable (via config/sorcc.ini) | Cellular data |
| Storage | 128GB+ SD card | OS + capture data |

### Config System

All tunables live in `config/sorcc.ini` (INI format). Sections:
`[general]`, `[lte]`, `[gps]`, `[kismet]`, `[dashboard]`, `[tailscale]`, `[pisugar]`, `[wifi]`, `[recon_tools]`.

- Schema: `sorcc/config_schema.py` вҖ” FieldSpec-based validation
- API: `sorcc/config_api.py` вҖ” read/write with file locking, backup, factory reset
- Factory defaults: `config/sorcc.ini.factory`
- Runtime config: `/opt/sorcc/config/sorcc.ini` (on Pi)

### API Endpoints

**Pages:**
- `GET /` вҖ” Dashboard SPA
- `GET /instructor` вҖ” Standalone instructor overview

**Status & Devices:**
- `GET /api/status` вҖ” System health (Kismet, modem, GPS, battery, Tailscale, hostname, device count, active profile)
- `GET /api/devices` вҖ” All WiFi/BT devices from Kismet
- `GET /api/target/{ssid}` вҖ” Hunt Mode: live RSSI for target SSID
- `GET /api/gps` вҖ” Current GPS position

**Config:**
- `GET /api/config/full` вҖ” Read all config (redacted sensitive fields)
- `POST /api/config/full` вҖ” Write config updates
- `POST /api/config/restore-backup` вҖ” Restore from .bak
- `POST /api/config/factory-reset` вҖ” Restore factory defaults
- `GET /api/config/export` вҖ” Download config as JSON
- `POST /api/config/import` вҖ” Upload JSON config

**Preflight:**
- `GET /api/preflight` вҖ” Run hardware/service checks, returns structured JSON

**Profiles:**
- `GET /api/profiles` вҖ” List RF mission profiles
- `GET /api/profiles/active` вҖ” Current active profile
- `POST /api/profiles/switch` вҖ” Switch profile (reconfigures Kismet sources)

**Export:**
- `GET /api/export/kml` вҖ” Export Kismet data as KML
- `GET /api/export/csv` вҖ” Export device list as CSV

## Development Guidelines

### Audience
The end users are soldiers вҖ” smart, motivated, but not Linux sysadmins. Every
script and UI must work without debugging. Error messages must tell the user
exactly what to do next. Silent failures are unacceptable.

### Scripts
- All bash scripts use `set -euo pipefail`
- Use color-coded output: `ok()`, `warn()`, `fail()`, `info()` helpers
- Every operation must be idempotent (safe to re-run)
- Auto-detect hardware вҖ” never hardcode device paths without fallbacks
- Use `$SORCC_USER` / `$SORCC_HOME` вҖ” never hardcode `/home/kali/`
- Read/write config from `config/sorcc.ini` вҖ” never hardcode carrier APNs

### Dashboard
- FastAPI + Jinja2 templates + vanilla JS SPA
- Three main tabs: Operations, Settings, Preflight
- Standalone instructor page at `/instructor`
- SORCC branding: green (#4a7c3f) / black / white / dark theme
- Design tokens in `variables.css` (follow Hydra's pattern)
- Must work on phones and tablets (responsive, large touch targets)
- No external CDN dependencies вҖ” all assets served locally
- Toast notifications for user feedback (success/info/error)
- System font stack вҖ” no web fonts

### Config Pattern
- Factory defaults in `config/sorcc.ini.factory`
- Runtime config at `/opt/sorcc/config/sorcc.ini`
- Typed validation via `config_schema.py` (FieldSpec dataclasses)
- File-locking writes via `config_api.py`
- Sensitive fields redacted in API responses (Kismet password, WiFi password)
- APN is configurable per-carrier вҖ” never hardcode to a single carrier

### RF Mission Profiles
- Defined in `profiles.json` at repo root
- Each profile specifies which Kismet data sources to activate
- Switched via web UI dropdown or POST `/api/profiles/switch`
- 5 profiles: WiFi Survey, Bluetooth Recon, TPMS Monitoring, ADS-B Aircraft, Full Spectrum

### Common Commands

```bash
# Run the installer
sudo bash scripts/sorcc-setup.sh

# Validate the install
bash scripts/sorcc-preflight.sh

# Headless boot setup
sudo bash scripts/sorcc-headless.sh --ssid "WiFiName" --password "pass"

# Capture old Pi configs
sudo bash scripts/sorcc-capture-old.sh

# Start Kismet manually
sudo kismet --no-ncurses

# Access the dashboard
# http://<pi-ip>:8080

# Check service status
systemctl status kismet sorcc-dashboard sorcc-boot

# Check modem/GPS
sudo mmcli -m 0
sudo mmcli -m 0 --location-get
```
